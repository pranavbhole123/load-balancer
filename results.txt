This is actually expected. Let me explain why.

Why there's no difference
Your backend is a Python http.server. It's single threaded — handles one request at a time. The bottleneck is the backend, not your connection pool.

Connection pooling helps when TCP handshake overhead is significant relative to request processing time. With a Python server that takes 20ms to respond, saving 1ms on handshake is noise.

Also you're testing locally — localhost. TCP handshake on localhost is ~0.1ms. On a real network it's 50-100ms. Connection pooling's benefit is proportional to network latency.

How to actually see the difference
You need a fast backend. Write one in Go:

go
// testserver/main.go
package main

import (
    "fmt"
    "net/http"
    "os"
)

func main() {
    port := os.Args[1]
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "backend %s", port)
    })
    http.ListenAndServe(":"+port, nil)
}
Run two instances:

bash
go run testserver/main.go 9090 &
go run testserver/main.go 9091 &
Now benchmark again with and without pooling. The Go backend responds in microseconds — handshake overhead becomes significant.

The numbers themselves matter less than understanding why. In your README write:

Connection pooling shows minimal improvement on localhost with slow backends.
Benefit is proportional to network round-trip time and backend response speed.
In production (cross-datacenter, fast backends) expected 3-5x improvement.
